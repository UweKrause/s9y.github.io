````
   * Fokus: Blog-System. kein CMS. Gewisse CMS-Kernfunktionen zwar "nice to have" aber kein Zentrum.
   * Einfach anpassbar, sowohl für Benutzer via Themes/Plugins/Optionen als auch für Entwickler über API
   * Niedrige Einstiegshürde, kurze Supportwege, „es funktioniert einfach“, offen für Impulse durch Nutzer und Entwickler.
   * Lang existierendes, gereiftes System mit Wert auf fortgeführte Nutzung und "Rücksicht" auf Benutzer
   * "Ein stabiles, sicheres, erweiterbares|flexibles|verlässliches Blog-System"
     * "Eine verlässliche, sichere und erweiterbare Blog-Software"
   * (Slogan)
   * Not mainstream since 2001
   * Ziele:
     * "Benutzerfreundlicher" (Bedienung so einfach wie möglich, Anpassung so einfach wie möglich, die Dinge abdecken können die beim Blogging relevant sind) ohne unnötige Komplexität.
     * Komponenten auf zukunftssichere Technologien anpassen, Kompatibilität und Sicherheit und bestehende Features bewahren
     * Klare, separierte und anpassbare Codeflows / Architektur
     * Nachvollziehbarkeit des Codes mit guter Verständlichkeit, für neue Entwickler ohne große Einarbeitungszeit "lesbar".
     * Funktionen so klar haben, dass keine Labels nötig sind
````

````
* Umgang mit Feature Requests und Bugs:
   * Primär erfolgen Support-Themen im Forum. Ein Foren-Account hat für einen Anwender höhere Priorität/Notwendigkeit als ein Github-Account
   * EIN BUG PRO ISSUE :-)
   * Github nur englisch, im Forum auch deutsche Meldungen möglich!
   * Konkrete Bugs (mit Fehlermeldung) können optimalerweise im GitHub-Issuetracker gemeldet werden. Sie können auch im Forum gemeldet werden und ein Entwickler fixt den Bug entweder so direkt, oder stellt es als Issue für den meldenden Nutzer ein
   * Neue Features werden nur von Entwicklern eingetragen in GitHub als Issue, wenn sie auch konkret umgesetzt werden können. Zu diskutierende features werden im Forum diskutiert, und erst wenn Konsens herrscht von einem Entwickler (oder vom Nutzer mit Github-Account) dann eingestellt werden. 
   * In Github gemeldete Features die als Diskussion im Forum geführt werden sollen werden sofort geschlossen und auf das Forum verwiesen
   * Im Forum gemeldete Bugs die als GitHub-Issue eröffent und geschlossen wurden sollten final einen kleinen "wurde gefixt"-Hinweis im Forum zurück erhalten.
   * Entwickler-Kollegen, die Bugs über Github melden, sollen ohne Forumsverweis willkommen geheißen werden
````
